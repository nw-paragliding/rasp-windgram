RUN PARAMETER DESCRIPTION

Parameters in file $BASEDIR/RASP/RUN/rasp.run.parameters.REGIONXYZ control the behavior of runs initiated by the command "run.rasp REGIONXYZ", where REGIONXYZ is the identifier (directory name) of regional grid setup data. This file is in Perl coding format, so parameter names beginning with "$" are for a single value whereas parameters beginning with "@" or "%" can contain multiple (array/hash) values. Note that all specified "hour" times below are UTC (GMT) not local time!

An overview of parameter specification is found at ParameterOverview.

Region-independent Parameters

  • @REGION_DOLIST
    Regions processed by a single run, normally set to "REGIONXYZ". It is possible to run two regional forecasts from the same NCEP "grib" data, so that it need be downloaded only once, by putting two region names here and then creating region-specific run parameters for both regions. (The corresponding regional run directories should have already been created.)

  • $GRIBFILE_MODEL
    NCEP driving model data ("grib files") to be used for RASP initialization and boundary conditions. For non-US runs this is either the "GFSA" or "GFSN" global model output (see GFSAvsGFSN) while for US runs it is normally the "ETA" regional model output. Note: the same argument must be used for parameters INIT_ROOT & LBC_ROOT in file $BASEDIR/WRF/wrfsi/domains/REGIONXYZ/static/wrfsi.nl and if a windowed run also in file $BASEDIR/WRF/wrfsi/domains/REGIONXYZ-WINDOW/static/wrfsi.nl.

  • $GRIB_LEFT_LON, $GRIB_RIGHT_LON,$GRIB_BOTTOM_LAT, $GRIB_TOP_LAT (GFSA only)
    These parameters are only utilized when $GRIBFILE_MODEL='GFSA'. They set the boundaries (longitude/latitude, in integer degrees) of the region extracted to a limited-area GFS file, which is then downloaded. Since this results in considerable saving of download time/cost, they should normally be set whenever GFS model downloads are requested. The boundaries must encompass the entire domain of the coarsest model domain. For further information see GFSAvsGFSN.

  • $LSEND
    Whether files will be created in optional website directory
    • 0 = images and data files produced only in local "RASP/RUN/OUT" subdirectory
    • 1 = copy images and data files to web directory using filenames pre-pended with "test"
    • 2 = copy images and data files to web directory using normal filenames
    • -1,-2 = send images and data files to remote server (NOT COMPLETED: calls script $BASEDIR/RASP/RUN/UTIL/rasp.multiftp, which is currently a test version using parameters which would need to be modified for other users (e.g. the remote server name) - if you get this script to work please let me know and I will adopt your version and generalize the script for others to use more easily. Alternatively you could write your own script to replace the existing one and I would adapt rasp.pl to utilize it.)
    • 3 = also do firstofday processing (NOT IMPLEMENTED)

  • $LSAVE (global)
    Whether files will be saved in archive directory
    • 0 = inhibits all archive saves to preserve disk space
    • 1 = saves data files at times specified by @{$SAVE_PLOT_HHMMLIST} - uses least disk space
    • 2 = saves images and data files at times specified by @{$SAVE_PLOT_HHMMLIST} - uses much more disk space than $LSAVE=1
    • 3 = saves images and data files at times specified by @{$SAVE_PLOT_HHMMLIST} and saves initial condition files to a storage directory

  • %LSAVE (hash array) - OPTIONAL
    Gives fine-scale control over saving of files to archive directory if needed. It is not normally set as the global $LSAVE flag meets most user needs. A more restrictive (smaller) LSAVE than the global value can be set for an individual hour and parameter by setting $LSAVE{REGIONXYZ}{parametername}{time}[IWINDOW]. For example, $LSAVE{PANOCHE}{blwindshear}{1500}[0]=0; will result in images and datafiles of the blwindshear parameter at 1500 lst not being saved for a non-windowed run when global $LSAVE=1. Setting values for numerous parameters and hours can be lengthy, but using simple perl programming can help. For example, the following is used by DrJack to save his archive files: setting global $LSAVE=2, the following reduces LSAVE to 1 for all except the "slice" plot, all pressure level plots, and all soundings at all requested output times for the non-windowed run (a similar loop is used for the windowed run): 
  foreach $param ( @{$PARAMETER_DOLIST{'PANOCHE'}} )
  {
  foreach $hhmm ( @{$SAVE_PLOT_HHMMLIST{'PANOCHE'}[0]} )
  {
  if( $param eq 'boxwmax' or $param =~ /press/ or $param =~ /sounding/  )
    { $LSAVE{'PANOCHE'}{$param}{$hhmm}[0] = 1 ; }
  }
  }

Overall Region-specific Parameters

  • $LRUN_WINDOW{'REGIONXYZ'}
    Controls "1-stage" vs "2-stage" behavior. Use 0 for a "1-stage" run and 2 for a "2-stage" run producing both "stage-1" and "stage-2" (window) forecasts. Setting this parameter to 1 will do a stage-2-only run using data from an existing previously-run 1-stage run - this is primarily for test runs but might have some operational utility.

  • @{$PARAMETER_DOLIST{'REGIONXYZ'}}
    Parameters to be plotted. The parameter list used in the supplied example region file $BASEDIR/RASP/RUN/rasp.run.parameters.PANOCHE (or SW_SOUTHAFRICA) contains all possible parameters. If desired, some can be eliminated for your region. The parameter names give the plots described at PlotParameterNames. Also, any desired sounding plot must be added to this list as described at ProgramUserChanges. Any variable available in the WRF output files can also be plotted, perhaps just for diagnostic purpose rather than for public presentation, as described at ProgramUserChanges.

  • $LOCALTIME_ID{'REGIONXYZ'}
    3 letter identification for identification of local time zone, e.g. "PST" for "Pacific Standard Time". Used only for printing information. Note that "daylight savings time" output is not produced by the model.

  • $LOCALTIME_ADJ{'REGIONXYZ'}
    Time zone adjustment. The difference in hours between local time zone and the UTC time zone. (Negative for US.)

  • $GRIBFILES_PER_FORECAST_PERIOD{'REGIONXYZ'}
    Number of GRIB files to be downloaded for a single forecast run. Normally this is 5 for a 12 hour forecast which will use GRIB data for the initialization and for later boundary conditions at 3 hour time intervals.

  • %GRIBFILE_DOLIST
    Individual GRIB files to be downloaded for the forecast runs, using format "iZ+f" where i is the GRIB model initialization time (in UTC) and f it's forecast period (in hours). For example, for a single 12 hour forecast which will use GRIB data from a 6Z ETA GRIB initialization for a RASP initialization at 15Z and for later RASP boundary conditions at 3 hour time intervals out to 3Z the next day, this parameter would be
    %GRIBFILE_DOLIST = ( 'REGIONXYZ' => [ '6Z+9','6Z+12','6Z+15','6Z+18','6Z+21', ], );
    i.e $GRIBFILES_PER_FORECAST_PERIOD=5 values all having the same GRIB initialization time but with forecast periods equally spaced by ($FORECAST_PERIODHRS/($GRIBFILES_PER_FORECAST_PERIOD-1))=3 hours. The forecast period must be larger for each successive file. Possible initialization times are 0Z, 6Z, 12Z, and 18Z. Additional forecast runs during a single day can be created by adding a sequence of GRIBFILES with a different GRIB initialization time see ProgramComplexRuns.

Time Parameters for Stage-1 Forecast Run

A RASP forecast "run" begins at its "initialization" time (which can only be 0Z, 3Z, 6Z, etc.) and runs for its specified forecast period (which must be a multiple of 3, and typically is 12 hours). The model's "equations of motion" must re-adjust to initialization imbalances, leading to forecast inaccuracies over the first 2-6 hours (the smaller/larger time corresponding to a smaller/larger grid spacing) - so forecast output should not be displayed for that "spinup" period. GRIB files at 3 hour intervals covering the specified forecast period are downloaded to provide initial and boundary conditions for the RASP run. The RASP initialization time equals the validation time of the first (earliest) GRIB file (where the GRIB validation time is the sum of the GRIB initialization time and GRIB forecast time). Note that in the following consistency between several different parameters is often required - for example, the difference between $DOMAIN1_STARTHH and $DOMAIN1_ENDHH must equal $FORECAST_PERIODHRS, the $DOMAIN1_STARTHH must equal the validation time of the first (earliest) GRIB file in a forecast period, etc. Please note that the script does not make many consistency checks!

  • $FORECAST_PERIODHRS{'REGIONXYZ'}[0]
    Forecast period covered by a single run, normally 12 hours.

  • $DOMAIN1_STARTHH{'REGIONXYZ'}[0]
    Two-digit start (model initialization) UTC hour for the coarse domain forecast, which must be an hour for which a downloaded NCEP "grib" file is available i.e. normally a GMT hour which is a multiple of 3. This time should be in the early morning, before or just after sunrise. ("Two-digit" means that a leading zero must be used if needed.)

  • $DOMAIN1_ENDHH{'REGIONXYZ'}[0]
    Two-digit end UTC hour (initialization hour + forecast period) for the coarse domain forecast. ("Two-digit" means that a leading zero must be used if needed. The end hour can be greater than 24, use the algebraic sum of initialization hour + forecast period, not the time of day.)

  • $DOMAIN2_START_DELTAMINS{'REGIONXYZ'}[0]
    Time between start of fine and coarse domain forecasts, in minutes - at present should be 0

  • $DOMAIN2_END_DELTAMINS{'REGIONXYZ'}[0]
    Time between end of fine and coarse domain forecasts, in minutes - at present should be 0

  • $DOMAIN3_START_DELTAMINS{'REGIONXYZ'}[0]
    Not used at present

  • $DOMAIN3_END_DELTAMINS{'REGIONXYZ'}[0]
    Not used at present

  • $DOMAIN1_TIMESTEP{'REGIONXYZ'}[0]
    Time step in seconds of the stage-1 coarse grid. See GridTimeStep for more information.

  • $BOUNDARY_UPDATE_PERIODHRS{'REGIONXYZ'}[0]
    Increment between boundary updates from GRID data, normally 3 hours.

  • $NDOWN_BOUNDARY_UPDATE_PERIODHRS{'REGIONXYZ'}[0]
    Always zero

  • @{$PLOT_IMAGE_SIZE{'REGIONXYZ'}[0]}
    Width and height of plot image, ala "600x600". First array value is for first (outer) domain images, second value for second (inner) domain images. To not plot a domain, use a blank value.

  • @{$PLOT_HHMMLIST{'REGIONXYZ'}[0]}
    Array containing times for which parameter plots will be created, in format "HHMM" where HH and MM are the UTC hour and minute in two-digit format. ("Two-digit" means that a leading zero must be used if needed.) This means that if you want to plot every hour from 10pm to 1am UTC the next day you will have to put in 2200, 2300, 0000, 0100 (these are not limited to the GRIBFILE_DOLIST times). Plots require creation of a WRF output file for that time, which by default is done for each hour; output for other times requires modification of parameter HISTORY_INTERVAL in file $BASEDIR/WRF/WRFV2/RASP/REGIONXYZ/namelist.template. Note that such plots require a model iteration be made for exactly that time, based on $DOMAIN1_TIMESTEP{'REGIONXYZ'}[0] - see GridTimeStep for more information, particularly if an expected plot does not appear!

  • @{$PLOT_LOOP_HHMMLIST{'REGIONXYZ'}[0]}
    Array containing times for which a loop of parameter plots will be created at the end of the stage-1 run, in format "HHMM" where HH and MM are the UTC hour and minute in two-digit format. ("Two-digit" means that a leading zero must be used if needed.) All times specified here must also be included in @{$PLOT_HHMMLIST{'REGIONXYZ'}[0]}. An entry of 'title' will add a title frame at the beginning of the loop.

  • @{$SAVE_PLOT_HHMMLIST{'REGIONXYZ'}[0]}
    Array containing times for which parameter plot images are stored, in format "HHMM" where HH and MM are the UTC hour and minute in two-digit format. When $LSAVE in file rasp.site.parameters.REGIONYZ is greater than 0, copies of the forecast plots will be saved for the specified times in directory $BASEDIR/RASP/SAVE/REGIONXYZ/yyyymmdd where yyyy is the year, mm the numeric month, and dd the day. To be saved, any such time must also be in $PLOT_HHMMLIST so that the plot gets created and $PLOT_IMAGE_SIZE must be non-blank for the desired domain. ("Two-digit" means that a leading zero must be used if needed.).

Time Parameters for Stage-2 ("Window") Forecast Run

These parameters are required only for a stage-2 "window" run, i.e. only when $LRUN_WINDOW{'REGIONXYZ'} > 0. Note that the parameters are identical to those of the preceeding section except "[1]" replaces "[0]" in the parameter names. When choosing times to display, note that "spinup" adjustments to the unbalanced initial state will occur during the first 1-3 hours (a smaller/larger time corresponding to a smaller/larger grid spacing) , producing forecast inaccuraries - so creating plots of times earlier than 1-2 hours after initialization is not recommended.

  • $FORECAST_PERIODHRS{'REGIONXYZ'}[1]
    Forecast period covered by a single run, normally 6 hours.

  • $DOMAIN1_STARTHH{'REGIONXYZ'}[1]
    Two-digit start (model initialization) UTC hour for the "windowd" coarse domain forecast, which must be an hour for which a downloaded NCEP "grib" file is available i.e. normally a GMT hour which is a multiple of 3. ("Two-digit" means that a leading zero must be used if needed.) While a start time equal to the stage-1 start hour [$DOMAIN1_STARTHH{'REGIONXYZ'}[0]] is allowed, the stage-2 start hour should be later than the stage-1 start hour since the coarser resolution grid has a longer spinup time.

  • $DOMAIN1_ENDHH{'REGIONXYZ'}[1]
    Two-digit end UTC hour (initialization hour + forecast period) for the coarse domain forecast. ("Two-digit" means that a leading zero must be used if needed. The end hour can be greater than 24, use the algebraic sum of initialization hour + forecast period, not the time of day.)

  • $DOMAIN2_START_DELTAMINS{'REGIONXYZ'}[1]
    Time between start of fine and coarse domain forecasts, in minutes - at present should be 0

  • $DOMAIN2_END_DELTAMINS{'REGIONXYZ'}[1]
    Time between end of fine and coarse domain forecasts, in minutes - at present should be 0

  • $DOMAIN3_START_DELTAMINS{'REGIONXYZ'}[1]
    Not used at present

  • $DOMAIN3_END_DELTAMINS{'REGIONXYZ'}[1]
    Not used at present

  • $DOMAIN1_TIMESTEP{'REGIONXYZ'}[1]
    Time step in seconds of the stage-2 coarse grid. See GridTimeStep for more information.

  • $BOUNDARY_UPDATE_PERIODHRS{'REGIONXYZ'}[1]
    Increment between boundary updates using previous stage-1 run - normally 1 hour.

  • $NDOWN_BOUNDARY_UPDATE_PERIODHRS{'REGIONXYZ'}[1]
    Must be same as $BOUNDARY_UPDATE_PERIODHRS{'REGIONXYZ'}[1] value above.

  • @{$PLOT_IMAGE_SIZE{'REGIONXYZ'}[1]}
    Width and height of plot image, ala "600x600". First array value is for first (outer) domain images, second value for second (inner) domain images. To not plot a domain, use a blank value.

  • @{$PLOT_HHMMLIST{'REGIONXYZ'}[1]}
    Array containing times for which parameter plots will be created, in format "HHMM" where HH and MM are the UTC hour and minute in two-digit format. ("Two-digit" means that a leading zero must be used if needed.) Note that such plots require a model iteration be made for exactly that time, based on $DOMAIN1_TIME{'REGIONXYZ'}[1] - see GridTimeStep for more information, particularly if an expected plot does not appear!

  • @{$PLOT_LOOP_HHMMLIST{'REGIONXYZ'}[1]}
    Array containing times for which a loop of parameter plots will be created at the end of the stage-2 run, in format "HHMM" where HH and MM are the UTC hour and minute in two-digit format. ("Two-digit" means that a leading zero must be used if needed.) All times specified here must also be included in @{$PLOT_HHMMLIST{'REGIONXYZ'}[1]}. An entry of 'title' will add a title frame at the beginning of the loop. Plots require creation of a WRF output file for that time, which by default is done at 30 minute intervals; output for other times requires modification of parameter HISTORY_INTERVAL in file $BASEDIR/WRF/WRFV2/RASP/REGIONXYZ-WINDOW/namelist.template. Note that such plots require a model iteration be made for exactly that time, based on $DOMAIN1_TIMESTEP{'REGIONXYZ'}[0] - see GridTimeStep for more information, particularly if an expected plot does not appear!

  • @{$SAVE_PLOT_HHMMLIST{'REGIONXYZ'}[1]}
    Array containing times for which parameter plot images are stored, in format "HHMM" where HH and MM are the UTC hour and minute in two-digit format. When $LSAVE in file rasp.site.parameters.REGIONYZ is greater than 0, copies of the forecast plots will be saved for the specified times in directory $BASEDIR/RASP/SAVE/REGIONXYZ/yyyymmdd where yyyy is the year, mm the numeric month, and dd the day. To be saved, any such time must also be in $PLOT_HHMMLIST so that the plot gets created and $PLOT_IMAGE_SIZE must be non-blank for the desired domain. ("Two-digit" means that a leading zero must be used if needed.).
